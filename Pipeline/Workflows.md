# Workflows

`Pipeline/Workflows.md` владеет рабочими процедурами проходов.

## Основной путь продукта
1. Принять запрос и назвать текущие фазу, рабочий поток и гейт.
2. Подтвердить источник задачи и домены-владельцы.
3. Проверить Git-состояние, синхронизироваться от `origin/develop` и открыть рабочую ветку.
4. Подтвердить текущие `ROAD/BACK/PLAN` или открыть новый плановый контур.
5. Прочитать только документы-владельцы по области прохода.
6. Зафиксировать реальные противоречия, гипотезы и блокирующие вопросы.
7. Выполнить минимальные изменения по владельцам смысла.
8. Выполнить проверки нужного уровня.
9. Закрыть журнальный и плановый контур.
10. Сделать смысловые локальные коммиты, один push и PR в `develop` через `gh`.

## Первый `product-start`
Фаза: `Подтверждение текущей истины`.
Рабочий поток: `Первый старт продукта`.
Гейт: `Гейт текущей истины`.

Процедура:
1. Не считать начальный запрос пользователя полным подтверждением текущей истины.
2. Проверить `Docs/Discovery/Interview.md`.
3. Если `Статус_текущей_истины: Не_подтверждена`, оставаться в аналитическом контуре.
4. Не заполнять ответы интервью догадками агента.
5. Фиксировать неявные предположения отдельно как гипотезы.
6. Задать пользователю только блокирующие вопросы.
7. Не записывать стек, зависимости и ограничения как подтверждённые требования без явного источника.
8. После явных ответов синхронизировать `Plans/*` и `Logs/*`.
9. Выполнить структурную проверку продукта.

## Предметный проход
Фаза: `Реализация`.
Рабочий поток: `Предметный проход`.
Гейт: `Гейт реализации`.

Процедура:
1. Убедиться, что текущая истина подтверждена.
2. Проверить, что область предметного прохода не смешана с bootstrap-заготовками.
3. Зафиксировать требования, ограничения и источник выбора стека.
4. Вносить изменения только в минимальный набор owner-artifacts.
5. Разделять документацию, код, тесты, проверки и журнальное закрытие.
6. Не открывать новые домены без правила, владельца, потребителя и проверки.
7. Закрыть проход проверками, фактами в журналах и PR через `gh`.

## Гейты
- `Гейт рабочей ветки` — содержательные правки запрещены в `main` и `develop`; рабочие ветки используют только типы `chore/`, `feature/`, `fix/`, `docs/`; первый аналитический проход bootstrap-created продукта рекомендует `chore/`, а `docs/` остаётся для обычных документационных проходов после снятия стартового гейта.
- `Гейт текущей истины` — bootstrap-created продукт не переходит к предметной реализации без явных ответов пользователя.
- `Гейт планирования` — проход с правками должен иметь `ROAD/BACK/PLAN` или явно зафиксированное отсутствие активного этапа.
- `Гейт зависимостей` — стек и зависимости принимаются только из допустимого источника.
- `Гейт проверки` — структурные, тестовые, запусковые и ручные проверки не смешиваются в один неясный статус; уровни и доказательства проверки описаны в `Docs/Technical/Verification.md`.
- `Гейт журнального закрытия` — результат фиксируется в `Logs/*` фактами, а не намерениями.
- `Гейт запроса на слияние` — PR создаётся через `gh`; GitHub connector не является основным маршрутом создания PR.

## Уровни проверок
- `Структура` — наличие обязательных файлов, доменов, ID и запретов на удалённые домены.
- `Тесты` — автоматические тесты кода, если код или тестируемая логика затронуты.
- `Запуск` — фактический запуск CLI, сервиса, GUI или smoke-route.
- `GUI-запуск` — отдельная проверка доступности GUI-модуля и среды.
- `Ручная проверка` — действие человека или агента, которое нельзя честно автоматизировать.

Если GUI или иной запуск нельзя проверить в текущей среде, результат записывается как `не проверено`, а не как успешная проверка.

## Журнальное закрытие
1. `Logs/ADRlog.md` обновляется для архитектурных, процессных и продуктово-договорных решений.
2. `Logs/ChangeLog.md` фиксирует только факт изменения и эффект.
3. `Logs/QualityLog.md` фиксирует фактически выполненные проверки и непроверенные зоны.
4. Старые исторические записи не переписываются без необходимости.
5. `ROAD/BACK/PLAN` закрываются после проверок, а не до них.
6. Если журнальный отчёт или финальный ответ сообщает, что `ROAD/BACK/PLAN` закрыты, фактические файлы `Plans/Roadmap.md`, `Plans/Backlog.md` и архивированный `Plan` должны подтверждать статус `Завершено`.

## PR-маршрут
1. Перед push повторить обязательные проверки.
2. Выполнить один final push рабочей ветки.
3. Проверить через `gh`, что для head-ветки нет открытого PR.
4. Создать PR в `develop` через `gh pr create`.
5. GitHub connector не используется для создания PR.
6. Если `gh` недоступен или не авторизован, выполнить push и вернуть ссылку или точные параметры для ручного создания PR.

## Смысловые коммиты
Если в блоке есть изменения, он фиксируется отдельным локальным коммитом:
- подтверждение текущей истины;
- планирование;
- документация;
- код и тесты;
- журнальное закрытие;
- служебная синхронизация.
