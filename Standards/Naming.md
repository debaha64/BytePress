# Naming

## Версии
Версии оформляются в формате `MAJOR.MINOR.PATCH`.

Текущая baseline-version `BytePress`: `0.1.0`.

Активные документы используют semver, а не метки вида `vN`, когда речь идёт о текущем operational contract системы.

Semver применяется к текущему operational contract `BytePress`, а не к rewrite истории журналов.

## ID сущностей
Числовая часть ID содержит 6 знаков.

Типовой префикс сущности сохраняется:
- `PLAN-000001`
- `BACK-000001`
- `ADR-000001`
- `CHG-000001`
- `PROF-000001`

## Имена файлов и путей
Имена файлов и путей строятся по семантике домена и не дублируют родительский контекст без необходимости.

`slug` оформляется в `kebab-case`.

Имя файла не должно дублировать родительский каталог.

## Имена plan-file
Целевое каноническое имя файла плана:

`Plans/PLAN-<NNNNNN>-<slug>.md`

Где:
- `PLAN` — типовой префикс plan-file;
- числовая часть содержит 6 знаков;
- `slug` оформляется в `kebab-case`.

До отдельной migration-задачи historical `BP-*` plan-files могут физически оставаться в `Plans/` как legacy/transitional слой.

## Категории доменов по filename policy
Домены делятся на три группы:
- серийные домены с нормализацией filename и внутреннего ID;
- гибридные домены с семантическим filename и нормализованным внутренним ID;
- singleton-домены с семантическим filename без перехода на ID в имени файла.

### Серийные домены
Для серийных доменов новые артефакты создаются только по новому контракту, а legacy-слой мигрируется отдельными проходами.

- `Plans/`: целевой формат `Plans/PLAN-<NNNNNN>-<slug>.md`, historical `BP-*` допустимы только как transitional legacy-слой до отдельной migration-задачи
- `Docs/Terms/`: целевой формат `Docs/Terms/TERM-<NNNNNN>-<slug>.md`

### Гибридные домены
Для гибридных доменов filename остаётся семантическим, а внутренний ID нормализуется до 6 знаков.

- `Profiles/` в репозитории `BytePress`: brand profiles хранятся с семантическими filenames по бренду;
- `product profile` хранится только в product repo.

### Singleton-домены
Singleton-домены не переводятся на ID в filenames.

- `Rules/`
- `Standards/`
- `Roles/`
- `Skills/`
- `Adapters/`
- `Memory/`
- `MCP/`
- аналогичные домены с семантическими singleton-файлами

## Политика фазной миграции
Миграция 4-значных ID в `BytePress` выполняется полностью, но фазами.

Принципы:
- новые артефакты создаются только по новому контракту;
- legacy-слой мигрируется отдельными управляемыми проходами;
- migration policy не означает немедленный массовый rewrite всех исторических артефактов;
- прямые зависимости домена нормализуются вместе с целевой фазой, а не заранее по всему репозиторию.

Historical logs подлежат миграции, но в отдельной поздней фазе после нормализации текущих рабочих слоёв.

## Рабочие ветки
Формат рабочей ветки: `<type>/<NNNNNN>-<slug>`.

Допустимые `type`:
- `feat`
- `fix`
- `chore`
- `docs`
- `refactor`
- `release`
- `hotfix`
