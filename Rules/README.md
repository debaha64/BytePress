# Rules

`Rules/` хранит обязательные проектно-специфичные ограничения `BytePress`.

## Назначение
- фиксировать запреты и обязательные условия;
- принимать обязательные нормы, перенесённые из прежнего слоя стандартов;
- давать проверяемые границы для агента, человека и инструментов.

## Границы
Правило отвечает на вопрос «что обязательно или запрещено». Рекомендации, которые не являются обязательными условиями, не должны жить в `Rules/*`. Технический договор живёт в `Docs/Technical/*`, а факты выполнения — в `Logs/*`.

Проверочный договор не дублируется в правилах: классы проверок, уровни, доказательства и подтверждение результата принадлежат `../Docs/Technical/Verification.md`.

## Правило именования
Путь задаёт домен, а имя файла задаёт предмет договора внутри домена. Файл правила называется по области договора, а не длинной формулировкой условия. Исторические файлы не переименовываются без отдельной причины.

## Карта артефактов
- `Source.md` — источник истины, граница правил и порядок договоров перед массовым наполнением (`RULE-000003`, `RULE-000005`, `RULE-000010`).
- `Security.md` — запрет секретов в Git (`RULE-000002`).
- `Domains.md` — границы доменов, временные outputs и запрет преждевременных доменов (`RULE-000004`, `RULE-000007`, `RULE-000011`).
- `Workflow.md` — строгость утверждений и основание планов (`RULE-000001`, `RULE-000008`).
- `Git.md` — PR через `gh` и смысловые локальные коммиты (`RULE-000013`, `RULE-000014`).
- `Logs.md` — журналы фиксируют только факты и значимые ADR (`RULE-000009`).
- `Dependencies.md` — выбор стека и зависимостей только из явного источника (`RULE-000012`).
- `Terms.md` — управление словарём и конфликтами терминов (`RULE-000006`).
- `Naming.md` — семантическое именование путей и файлов (`RULE-000015`).
