## RULE-000016 — Безопасные типы рабочих веток
ID: RULE-000016
Название: Безопасные типы рабочих веток
Статус: Принято
Связи: TERM-000021, Pipeline/Workflows.md
Источник: Исправляющий проход после полевого теста продукта
Дата_создания: 2026-04-30
Дата_изменения: 2026-04-30
Обязательность: Обязательно

### Правило
Рабочая ветка создаётся только в формате `<type>/<NNNNNN>-<slug>` с безопасным типом `chore/`, `feature/`, `fix/` или `docs/`.

Первый аналитический проход bootstrap-created продукта использует рекомендуемый тип `chore/`.

`docs/` используется только для обычных документационных проходов после снятия стартового гейта текущей истины.

### Запрет
`product/` не является рекомендуемым или допустимым типом рабочей ветки.

### Полевые тесты
Перед повторным тестом продуктового репозитория нужно удалить старые локальные рабочие ветки, если они больше не нужны.

## RULE-000013 — Маршрут запроса на слияние через gh
ID: RULE-000013
Название: Маршрут запроса на слияние через gh
Статус: Принято
Связи: Pipeline/Workflows.md
Источник: Запрос владельца от 2026-04-29
Дата_создания: 2026-04-29
Дата_изменения: 2026-04-30
Обязательность: Обязательно

### Правило
Запрос на слияние в `develop` создаётся через `gh`.

### Обязательные условия
- перед запросом на слияние рабочая ветка должна быть отправлена в remote;
- перед созданием запроса на слияние нужно проверить отсутствие открытого запроса на слияние для рабочей ветки;
- GitHub connector не используется как основной маршрут создания запроса на слияние;
- если `gh` недоступен или не авторизован, агент выполняет отправку ветки и возвращает человеку ссылку или точные параметры для ручного создания запроса на слияние.

### Граница
GitHub connector может использоваться для чтения контекста, если это не спорит с задачей, но не для создания запроса на слияние.

## RULE-000014 — Смысловые фиксации
ID: RULE-000014
Название: Смысловые фиксации
Статус: Принято
Связи: Pipeline/Workflows.md
Источник: Запрос владельца от 2026-04-29
Дата_создания: 2026-04-29
Дата_изменения: 2026-04-30
Обязательность: Обязательно

### Правило
Широкий проход не должен закрываться одной общей фиксацией, если в нём есть разные смысловые блоки.

### Обязательные блоки
Если блок содержит изменения, он фиксируется отдельной локальной фиксацией:
- подтверждение текущей истины;
- планирование;
- документация;
- код и тесты;
- журнальное закрытие;
- служебная синхронизация.

### Граница
Фиксации остаются локальными до финальных проверок. Отправка ветки выполняется один раз в конце прохода.
