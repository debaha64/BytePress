## RULE-000016 — Безопасные типы рабочих веток
ID: RULE-000016
Название: Безопасные типы рабочих веток
Статус: Принято
Связи: TERM-000021, Pipeline/Workflows.md
Источник: Исправляющий проход после полевого теста продукта
Дата_создания: 2026-04-30
Дата_изменения: 2026-04-30
Обязательность: Обязательно

### Правило
Рабочая ветка создаётся только в формате `<type>/<NNNNNN>-<slug>` с безопасным типом `chore/`, `feature/`, `fix/` или `docs/`.

Первый аналитический проход bootstrap-created продукта использует рекомендуемый тип `chore/`.

`docs/` используется только для обычных документационных проходов после снятия стартового гейта текущей истины.

### Запрет
`product/` не является рекомендуемым или допустимым типом рабочей ветки.

### Полевые тесты
Перед повторным тестом продуктового репозитория нужно удалить старые локальные рабочие ветки, если они больше не нужны.

## RULE-000013 — Маршрут запроса на слияние через gh
ID: RULE-000013
Название: Маршрут запроса на слияние через gh
Статус: Принято
Связи: Pipeline/Workflows.md
Источник: Запрос владельца от 2026-04-29
Дата_создания: 2026-04-29
Дата_изменения: 2026-04-30
Обязательность: Обязательно

### Правило
Запрос на слияние в `develop` создаётся через `gh`.

### Обязательные условия
- перед запросом на слияние рабочая ветка должна быть отправлена в remote;
- перед созданием запроса на слияние нужно проверить отсутствие открытого запроса на слияние для рабочей ветки;
- GitHub connector не используется как основной маршрут создания запроса на слияние;
- если `gh` недоступен или не авторизован, агент выполняет отправку ветки и возвращает человеку ссылку или точные параметры для ручного создания запроса на слияние.

### Граница
GitHub connector может использоваться для чтения контекста, если это не спорит с задачей, но не для создания запроса на слияние.

## RULE-000017 — Release и post-release sync
ID: RULE-000017
Название: Release и post-release sync
Статус: Принято
Связи: Pipeline/Workflows.md, Setup_Guide.md, Docs/Technical/Platform_Contracts.md
Источник: Post-release проход `0.3.0`
Дата_создания: 2026-05-18
Дата_изменения: 2026-05-18
Обязательность: Обязательно

### Правило
Обычные рабочие проходы доставляются через рабочую ветку и запрос на слияние в `develop`.

Release branch создаётся от актуального `develop` в формате `release/*`. Release PR открывается только из `release/*` в `main`.

Tag выпуска создаётся только после подтверждённого merge release PR в `main` и должен указывать на фактический commit выпуска в `main`.

Post-release sync возвращает `develop` к содержимому подтверждённого выпуска. Если force-push в защищённый `develop` запрещён, допустим sync через отдельную рабочую ветку и запрос на слияние в `develop`.

### Squash merge
После squash merge и post-release sync `main` и `develop` могут совпадать по дереву, но отличаться по истории.

Это не считается дефектом, если текущий договор требует совпадение содержимого выпуска, а не ancestor-отношение между ветками.

Требование ancestry между `main` и `develop` является отдельным решением о merge strategy и не внедряется молча.

## RULE-000022 — Гейт перед отправкой ветки и запросом на слияние
ID: RULE-000022
Название: Гейт перед отправкой ветки и запросом на слияние
Статус: Принято
Связи: Pipeline/Workflows.md, PLAN-000106
Источник: PLAN-000106
Дата_создания: 2026-05-22
Дата_изменения: 2026-05-22
Обязательность: Обязательно

### Правило
Перед `push` агент обязан проверить Git-состояние, текущий diff, согласованность планового контура, связанные артефакты по области прохода и обязательные проверки.

Минимальный набор проверок перед отправкой ветки: `git diff --check` и `python3 Tools/bp_lint.py --repo .`.

Перед созданием запроса на слияние рабочая ветка должна быть отправлена в remote, обязательные проверки должны быть пройдены, а описание PR должно содержать пакет доказательств: использованные ID, изменённые артефакты, commit-ы, проверки, состояние `ROAD/BACK/PLAN`, риски и ограничения.

Запрос на слияние создаётся через `gh`.

### Нарушение
Отправка ветки без сверки Git-состояния, diff, планового контура, связанных артефактов или обязательных проверок.

Создание PR не через `gh`, создание PR до отправки рабочей ветки в remote или без пакета доказательств в описании.

## RULE-000014 — Смысловые фиксации
ID: RULE-000014
Название: Смысловые фиксации
Статус: Принято
Связи: Pipeline/Workflows.md, PLAN-000105
Источник: Запрос владельца от 2026-04-29
Дата_создания: 2026-04-29
Дата_изменения: 2026-05-22
Обязательность: Обязательно

### Правило
Широкий проход не должен закрываться одной общей фиксацией, если в нём есть разные смысловые блоки.

Локальная фиксация отражает завершённый смысловой шаг из `PLAN-*`, а не случайный набор файлов. У шага должен быть проверяемый результат: изменённый договор, код, тест, журнал, плановый статус или явно подтверждённое отсутствие изменений в проверенном блоке.

Если смысловой блок не менялся, искусственная фиксация не создаётся. Проверка блока отражается в плане или журнале без пустого commit.

### Обязательные блоки
Если блок содержит изменения, он фиксируется отдельной локальной фиксацией:
- подтверждение текущей истины;
- планирование;
- документация;
- код и тесты;
- журнальное закрытие;
- служебная синхронизация.

### Граница
Фиксации остаются локальными до финальных проверок. Отправка ветки выполняется один раз в конце прохода.

После сбоя агент возобновляет работу от последнего проверенного состояния: текущая ветка, локальные фиксации, статус `PLAN-*`, журнальные факты и уже выполненные проверки. Повторная фиксация уже завершённого смыслового шага не создаётся без нового изменения этого шага.

### Широкий исправляющий проход
Если в проходе есть соответствующие изменения, отдельными локальными фиксациями закрепляются:
- аналитическое подтверждение текущей истины;
- открытие или существенная синхронизация планового контура;
- реализация;
- документально-журнальное закрытие.

Искусственные фиксации не создаются, если блок фактически не менялся.
