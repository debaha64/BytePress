# Pipeline

## Назначение
Конвейер в `BytePress` — это формальная модель прохождения изменения через фазы, входы, выходы, точки утверждения и обязательные артефакты.

## Базовый золотой путь `0.1.0`
`Обсуждение -> Интервью -> Исследование -> Требования -> Пакет основания -> Roadmap -> Backlog -> План -> Утверждение -> Реализация -> Проверка -> Выпуск -> Передача -> Поддержка`

## Роль конвейера в системе
Конвейер не хранит рабочие данные сам по себе. Он задаёт:
- порядок фаз;
- требования к переходам;
- обязательные артефакты;
- точки ручного утверждения;
- привязку результата фазы к домену хранения.

## Привязка фаз к доменам
- аналитические фазы заполняют `Docs/`;
- фазы `Roadmap`, `Backlog` и `Plan` работают через `Plans/`;
- исполнение ведётся через `Runtime/` с опорой на утверждённый `Plan`;
- подтверждённые факты фиксируются в `Logs/`;
- правила переходов и строгость фазы ограничиваются `Profiles/` и `Rules/`.

## Точки утверждения
В `Default` профиле `0.1.0` ручное утверждение требуется на каждой фазе. Этот режим считается параметром профиля, а не вечным универсальным правилом.

## Артефакты перехода
- после `Требований` должен существовать пакет основания;
- после `Backlog` должен существовать backlog-элемент, пригодный для планирования;
- после `Plan` должен существовать утверждённый план с шагами и DoD;
- после `Проверки` должны существовать подтверждённые записи в журналах.

## Техническая граница
`Pipeline/` описывает движение работы, но не заменяет:
- `Plans/` как хранение утверждённого намерения;
- `Logs/` как хранение фактов;
- `Docs/` как хранение знания;
- `Runtime/` как хранение оперативного состояния.
