# Usage Scenarios

## Назначение
Этот документ фиксирует минимальные базовые сценарии использования `BytePress` как системы.

Сценарии здесь остаются короткими и направляют к owner-documents вместо того, чтобы дублировать их.

## Сценарий 1. Войти в существующий репозиторий BytePress
- Прочитать `../../README.md` как карту системы.
- Проверить human operating mode через `Operating_Mode.md`.
- Если среда не готова, пройти `../../Setup_Guide.md`.
- Запустить `python3 ../../Tools/bp_lint.py --repo .` из корня репозитория, чтобы убедиться, что базовая структура валидна.

## Сценарий 2. Запустить новый pass с агентом
- Найти текущий stage/task/pass в `../../Plans/Roadmap.md`, `../../Plans/Backlog.md` и current `Plan`.
- Сформулировать узкий scope и ограничения для pass.
- Направить агента на задачу, опираясь на repo contracts, а не на ручной пересказ всего operating loop.
- Проверить результат по checks, governance-сверке и PR-flow, которые агент обязан пройти по `../../AGENTS.md`.

## Сценарий 3. Использовать BytePress как файловую систему знаний и планирования
- Смотреть `Docs/*` как долговременное знание по доменам.
- Смотреть `Plans/*` как канонический planning-layer.
- Смотреть `Logs/*` как слой фактов, изменений и quality evidence.
- Смотреть `Tools/*` только как executable support поверх уже утверждённых contracts.

## Сценарий 4. Понять, куда идти по типу вопроса
- setup, среда, зависимости: `../../Setup_Guide.md`;
- operating canon агента: `../../AGENTS.md`;
- техническая модель и контракты: `../Technical/README.md`;
- planning-state и текущая работа: `../../Plans/README.md`;
- продуктовая рамка: `../Product/README.md`.

## Граница сценариев
- Эти сценарии не являются полным user manual.
- Они задают только минимальный first-usable layer для человека.
- Расширение user-layer требует отдельного pass внутри `ROAD-000012`, а не неявного роста этого документа.
