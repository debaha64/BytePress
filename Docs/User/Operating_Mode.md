# Operating Mode

## Назначение
Этот документ фиксирует human operating mode в `BytePress`.

Он не описывает operating loop агента и не подменяет `AGENTS.md`; его задача — показать человеку, как правильно входить в систему и где проходит граница между human steering и agent execution.

## Базовый режим
- Человек задаёт направление, scope и критерии результата.
- Агент исполняет работу внутри repo contracts.
- Канонический operating canon агента закреплён в `../../AGENTS.md`.
- Канонический planning-state закреплён в `../../Plans/*`.

## Что делает человек
- определяет цель pass или задачи;
- задаёт ограничения и критерии приёмки;
- проверяет, что работа идёт в нужном stage/task/pass;
- оценивает результат и решает, достаточно ли он покрывает ожидаемый outcome;
- принимает решение о следующем pass, а не оставляет это неявным догадкам агента.

## Что не должен делать человек
- не использовать длинный ручной промпт как замену repo contracts, если owner-documents уже существуют;
- не ожидать, что `Docs/User/*` заменит setup guide, technical contracts или planning artifacts;
- не смешивать human steering с попыткой вручную переписать operating canon агента в каждом запросе.

## Рабочая модель взаимодействия
1. Человек читает карту системы и определяет текущую цель.
2. Человек направляет агента на конкретный pass или задачу.
3. Агент читает owner-documents и выполняет работу через task-ветку, checks и PR-flow.
4. Человек принимает результат, замечания и решение о следующем шаге.
